====== USB Connection Problem ======
Created Thursday 05 February 2026

===== 📋 EXECUTIVE SUMMARY =====

**Objective:** Establish a reliable USB connection between Raspberry Pi Zero W and Kali Linux PC as a backup to WiFi.

**Result:** ❌ **USB connection is NOT RELIABLE due to kernel driver bug**

**Root Cause:** The `cdc_ether` driver in Kali Linux kernel has a known bug with Raspberry Pi Zero W USB gadget mode, causing `NETDEV WATCHDOG` timeout errors.

**Current Status:** 
* ✅ WiFi connection via Starlink working perfectly (`argos.local` / `192.168.1.203`)
* ✅ Full system backup created (`argos_pi_backup_20260205.img.gz`)
* ❌ USB connection abandoned due to driver incompatibility
* 🔄 Alternative solution identified: TP-Link router "iotrac" for isolated network

---

===== 🔍 PROBLEM INVESTIGATION =====

==== Initial Symptoms ====
* USB connection worked initially after manual configuration
* Connection failed after Pi reboot or USB cable reconnection
* `ping 192.168.7.2` showed "Destination Host Unreachable" from PC
* Interface `eth1` showed `NO-CARRIER` state on PC

=== Kernel Error Logs: ===

{{{code: lang="bash" linenumbers="False"

=== Interface State: ===

{{{code: lang="bash" linenumbers="False"

**Diagnosis:** The `cdc_ether` driver fails to maintain a stable connection with Pi Zero W USB gadget, causing transmission timeouts and link drops.

===== 🧪 TESTS PERFORMED =====

==== Test 1: Cable and Port Verification ====
**Objective:** Verify USB cable supports data transfer and port is USB 2.0

**Method:**
{{{code: lang="bash" linenumbers="False"
# Map all USB buses and speeds
for bus in /sys/bus/usb/devices/usb*; do
    busnum=$(basename $bus | sed 's/usb//')
    speed=$(cat $bus/speed)
    echo "BUS $busnum: $speed Mbps"
done

# Check Pi connection
lsusb -t | grep -A2 "Bus 003"
}}}

**Result:** ✅ **PASS**
* Cable: Data cable confirmed (device detected)
* Port: USB 2.0 (480 Mbps) - correct for Pi Zero W
* Detection: Pi appears as `RNDIS/Ethernet Gadget` on Bus 003

---

==== Test 2: Driver Module Verification ====
**Objective:** Verify correct kernel modules are loaded

**Method:**
{{{code: lang="bash" linenumbers="False"
# On Pi
lsmod | grep -E "dwc2|g_ether"

# On PC
lsmod | grep -E "cdc_ether|rndis"
}}}

**Result:** ✅ **PASS**
* Pi: `dwc2` and `g_ether` modules loaded
* PC: `cdc_ether` driver loaded and bound to device
* **BUT:** Driver has timeout bug

---

==== Test 3: Alternative Driver Configuration ====
**Objective:** Test `g_ether` direct mode with `dr_mode=peripheral`

**Method:**
{{{code: lang="bash" linenumbers="False"
# On Pi - modify config.txt
sudo sed -i 's/dtoverlay=dwc2$/dtoverlay=dwc2,dr_mode=peripheral/' /boot/firmware/config.txt
sudo reboot

# On PC - verify detection
lsusb -t
ip link show eth1
}}}

**Result:** ❌ **FAIL**
* Device still detected with `cdc_ether` driver
* Interface still shows `NO-CARRIER`
* `NETDEV WATCHDOG` errors persist

**Conclusion:** Problem is in PC's kernel driver, not Pi configuration.

---

===== 🚫 FINAL CONCLUSION =====

==== Why USB Connection Failed ====

1. **Hardware:** ✅ All components working correctly
2. **Pi Configuration:** ✅ Correct setup, `usb0` interface functional
3. **PC Detection:** ✅ Device detected, driver loaded
4. **PC Driver:** ❌ **`cdc_ether` driver has known bug with Pi Zero W**

**The `cdc_ether` driver in Kali Linux kernel cannot maintain stable transmission with Raspberry Pi Zero W USB gadget mode.**

This is a **known kernel bug**, not a configuration issue.

===== 🔄 ALTERNATIVE SOLUTION: TP-LINK ROUTER "IOTRAC" =====

==== Overview ====

A TP-Link router named "iotrac" is available as an alternative backup connection method. This router can provide an isolated, stable network between the PC and Raspberry Pi, independent of internet connectivity.

**Advantages:**
* ✅ Dedicated network (no interference from other devices)
* ✅ Static IP configuration possible
* ✅ No dependency on Starlink or internet
* ✅ Reliable Ethernet/WiFi connection
* ✅ No kernel driver bugs

**Disadvantages:**
* ⚠️ No internet access on Pi when using this network
* ⚠️ Requires switching networks on PC to access Pi

=== Step 2: Configure Router ===

**Recommended Settings:**
1. **DHCP Server:** Enabled
   * Range: 192.168.0.100 - 192.168.0.200
   * Gateway: 192.168.0.1
   
2. **Static DHCP Reservation for Pi:**
   * MAC Address: `B8:27:EB:29:5E:5F` (Pi's MAC)
   * Reserved IP: `192.168.0.2`
   
3. **WiFi Settings:**
   * SSID: `iotrac` (keep existing)
   * Security: WPA2-PSK
   * Password: (set/verify password)

---

=== Step 3: Configure Pi for iotrac Network ===

**Add iotrac WiFi connection on Pi:**
{{{code: lang="bash" linenumbers="False"
# Connect to Pi via Starlink first
ssh pi@argos.local

# Add iotrac connection
sudo nmcli con add type wifi ifname wlan0 con-name "iotrac" ssid "iotrac"

# Set password
sudo nmcli con modify "iotrac" wifi-sec.key-mgmt wpa-psk wifi-sec.psk "<password>"

# Configure static IP
sudo nmcli con modify "iotrac" ipv4.addresses "192.168.0.2/24"
sudo nmcli con modify "iotrac" ipv4.gateway "192.168.0.1"
sudo nmcli con modify "iotrac" ipv4.dns "192.168.0.1,8.8.8.8"
sudo nmcli con modify "iotrac" ipv4.method manual

# Set autoconnect with lower priority than Starlink
sudo nmcli con modify "iotrac" connection.autoconnect yes
sudo nmcli con modify "iotrac" connection.autoconnect-priority 50

# Test connection
sudo nmcli con up "iotrac"
ping -c 3 192.168.0.1
}}}

**Note:** Starlink connection should have priority 100, iotrac priority 50, so Pi prefers Starlink when available.
