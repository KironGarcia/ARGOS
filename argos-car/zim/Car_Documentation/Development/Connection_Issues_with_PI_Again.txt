==== ABOUT STARLINK IP: ====

**YES, you can be calm.** Starlink routers (and home routers in general) maintain **much more stable** IPs than cell phone hotspots because:

✅ **DHCP with reservations:** The router "remembers" which IP it gave to each device by its MAC address  
✅ **Long lease time:** IPs are automatically renewed for days/weeks  
✅ **Fixed network:** The `192.168.1.x` range doesn't change like in mobile hotspots

**Probability of change:** Very low (only if you reset the router or the Raspberry is off for a long time).

===== 📋 COMPARATIVE DIAGNOSTIC SUMMARY =====

==== INITIAL PROBLEM (First Time - January 20) ====

=== Symptoms: ===
* Raspberry Pi did not allow SSH access
* Appeared connected by MAC but no access
* Static IP in incorrect range

=== Problems Found: ===
1. ❌ **SSH disabled** - `/boot/firmware/ssh` did not exist
2. ❌ **Network configured as hidden** - `hidden=true` in NetworkManager
3. ❌ **Password in encrypted hash** - Instead of plain text
4. ❌ **Static IP in incorrect range** - `10.154.149.x` when hotspot used another range

=== Solutions Applied (First Time): ===
{{{code: lang="bash" linenumbers="False"
# Enable SSH
sudo touch /media/kiron/rootfs/boot/firmware/ssh

# Edit WiFi configuration
sudo nano /media/kiron/rootfs/etc/NetworkManager/system-connections/preconfigured.nmconnection
# - Remove hidden=true
# - Change psk to plain text
# - Update IP to correct range
}}}

=== Result: === 
✅ Successful WiFi connection

---

==== RECURRING PROBLEM (Today - February 5) ====

=== Symptoms: ===
* Raspberry Pi was NOT appearing connected on the network (neither on Starlink nor on hotspot)
* Did not respond to SSH from any IP
* Network change (hotspot → Starlink)

=== Problems Found (NEW): ===
1. ❌ **NetworkManager with `managed=false`** - NetworkManager was not managing interfaces
2. ❌ **CORRUPTED config.txt** - File full of null bytes (`^@`)
3. ❌ **Malformed cmdline.txt** - Missing space between `rootwait` and `modules-load`
4. ❌ **Outdated IP range** - Configured for `172.25.54.x` (hotspot) when `192.168.1.x` (Starlink) was needed

=== Solutions Applied (Today): ===

**1. Change NetworkManager to managed:**
{{{code: lang="bash" linenumbers="False"
sudo nano /media/kiron/rootfs/etc/NetworkManager/NetworkManager.conf
# Change: managed=false → managed=true
}}}

**2. Rebuild corrupted config.txt:**
{{{code: lang="bash" linenumbers="False"
sudo nano /media/kiron/bootfs/config.txt
# Recreate complete standard Raspberry Pi configuration
}}}

**3. Fix cmdline.txt:**
{{{code: lang="bash" linenumbers="False"
sudo nano /media/kiron/bootfs/cmdline.txt
# Add space: rootwait modules-load=dwc2,g_ether
}}}

**4. Change to automatic DHCP:**
{{{code: lang="bash" linenumbers="False"
sudo nano /media/kiron/rootfs/etc/NetworkManager/system-connections/preconfigured.nmconnection
# [ipv4]
# method=auto  # Instead of static IP
}}}

**5. Update WiFi credentials to Starlink:**
{{{code: lang="ini" linenumbers="False"
ssid=KironStarlink
psk=124875369.lapergunta200
}}}

=== Result: === 
✅ Successful WiFi connection with DHCP

===== 🎯 LESSONS LEARNED =====

==== Recurring Problems: ====
1. **cmdline.txt** always has the space error between `rootwait` and `modules-load`
2. **Static IPs** cause problems when the network changes
3. **Files can get corrupted** (config.txt was corrupted this time)

==== Best Practices: ====
✅ **Use automatic DHCP** instead of static IP (more flexible)  
✅ **Verify `managed=true`** in NetworkManager.conf  
✅ **Always verify config.txt and cmdline.txt** after problems  
✅ **Use Starlink/fixed router** instead of mobile hotspot (more stable)  
✅ **Keep backup of working configurations**
