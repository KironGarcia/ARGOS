====== New Future for ARGOS ======
Created Thursday 09 January 2025

===== Project Evolution Overview =====

ARGOS is transitioning from a simple autonomous car to an intelligent robot with personality, dual-brain architecture, and advanced AI capabilities.

===== Critical Motor Problem Identified =====

==== Original Configuration ====
* **Motors**: DC 3.7V (not 12V-24V as initially documented)
* **Problem**: Insufficient torque for car weight
* **Symptom**: Motors cannot move the car (no startup power)
* **Root cause**: Underpowered motors for the chassis mass

==== Solution Implemented ====
* **New motor**: JGA25 12V DC gearmotor
* **New battery**: 12V 10900mAh
* **Status**: Components ordered, pending arrival
* **Expected result**: Adequate torque and power for movement

===== Architecture Redesign: Dual Brain System =====

The project now implements a symbiotic dual-controller architecture inspired by biological nervous systems, where specialized processors handle different cognitive loads.

==== ESP32 (Secondary Brain - Peripheral Nervous System) ====
* **Role**: Real-time sensor processing and peripheral control
* **Responsibilities**:
  - HC-SR04 ultrasonic sensor (obstacle detection)
  - IMU sensor (orientation, tilt, acceleration)
  - Servo motor control (steering + antenna expressions)
  - Audio system (microphone + speaker)
  - LCD 1602 display (text messages)
  - CAN bus peripheral management
* **Power source**: Rechargeable AA batteries 1.2V
* **Communication**: UART, I2C, WiFi, Bluetooth

==== Raspberry Pi Zero W (Primary Brain - Central Nervous System) ====
* **Role**: High-level processing and AI decision-making
* **Responsibilities**:
  - OV7670 camera (computer vision, object detection)
  - AI/ML algorithms (voice recognition, image processing)
  - Navigation logic (path planning, decision trees)
  - Data storage (logs, telemetry, learning data)
  - System coordination (orchestrating ESP32 commands)
* **Power source**: Powerbank 5V 10,000mAh
* **Communication**: UART to ESP32, WiFi for remote access

===== Symbiotic Communication Protocol =====

==== Physical Interfaces ====
* **UART (Serial)**: Primary data exchange (sensor data, commands)
* **I2C**: Shared peripheral access (LCD, IMU)
* **WiFi**: Telemetry and remote control
* **CAN bus**: Organized peripheral control

==== Data Flow Architecture ====
{{{code: lang="text" linenumbers="True"
ESP32 (Real-time Layer)
├── Sensors → Raw data processing
├── Actuators → Immediate response
└── UART → Send processed data to Raspberry Pi

Raspberry Pi (Intelligence Layer)
├── UART → Receive sensor data from ESP32
├── Camera → Computer vision processing
├── AI Engine → Decision making
└── UART → Send high-level commands to ESP32
}}}

===== Power Distribution System =====

==== Three Independent Power Sources ====

===== Source 1: ESP32 Power Supply =====
* **Battery type**: Rechargeable AA 1.2V (4x = 4.8V)
* **Components powered**: ESP32 only
* **Consumption**: ~80-240mA (idle to active)
* **Runtime**: ~6-8 hours (2500mAh batteries)

===== Source 2: Raspberry Pi Power Supply =====
* **Battery type**: Powerbank 5V 10,000mAh
* **Components powered**: Raspberry Pi Zero W + Camera
* **Consumption**: ~200-500mA (idle to camera active)
* **Runtime**: ~10-15 hours

===== Source 3: Motor Power Supply =====
* **Battery type**: 12V 10900mAh
* **Components powered**: L298N + JGA25 motors + servos
* **Consumption**: ~500-2000mA (variable with load)
* **Runtime**: ~5-10 hours (depending on usage)

==== Power Architecture Diagram ====
{{{code: lang="text" linenumbers="True"
Battery 1 (AA 1.2V x4) → ESP32 → Sensors + LCD + Audio
Battery 2 (Powerbank 5V) → Raspberry Pi → Camera + AI Processing
Battery 3 (12V 10900mAh) → L298N → Motors + Servos

Common Ground: All systems share GND reference via Raspberry Pi
}}}

==== Advantages of Independent Power Sources ====
* **Isolation**: Electrical fault in one system doesn't affect others
* **Efficiency**: Each system operates at optimal voltage
* **Redundancy**: ESP32 can continue basic operations if Raspberry Pi fails
* **Scalability**: Easy to upgrade individual power systems
* **Safety**: Motor power spikes don't affect logic circuits

===== ARGOS Personality System =====

ARGOS is designed as an intelligent character with emotional responses, inspired by R2-D2's iconic communication style.

==== Communication Methods ====

===== Visual Communication (LCD 1602) =====
* **Primary interface**: Text messages on 16x2 character display
* **Message types**:
  - Status updates ("SCANNING AREA")
  - Emotional states ("I'M HAPPY!")
  - Responses to commands ("UNDERSTOOD")
  - Warnings ("OBSTACLE AHEAD")
  - Questions ("WHICH WAY?")

===== Audio Communication (Speaker) =====
* **Style**: R2-D2 inspired beeps and chirps (not speech)
* **Sound patterns**:
  - Happy: Rising tone sequences
  - Sad: Descending tone sequences  
  - Alert: Rapid alternating tones
  - Confused: Random tone pattern
  - Acknowledgment: Short beep
  - Error: Descending buzzer sound

===== Physical Expression (Servo Antenna) =====
* **Component**: Second servo motor (not used for steering)
* **Function**: Movable antenna acting as "ear" or "tail"
* **Expression mapping**:
  - **Happy**: Antenna up (90°)
  - **Sad**: Antenna down (0°)
  - **Alert**: Antenna oscillating rapidly
  - **Curious**: Antenna tilting side to side
  - **Thinking**: Antenna slow circular motion
  - **Excited**: Antenna rapid up-down movement

==== Emotional State Machine ====
{{{code: lang="text" linenumbers="True"
User Command → Voice Recognition (Raspberry Pi)
             ↓
Decision: Command Valid?
             ├── YES → State = HAPPY
             │         ├── LCD: "YES SIR!"
             │         ├── Speaker: Happy beeps
             │         └── Antenna: UP position
             │
             └── NO → State = CONFUSED
                       ├── LCD: "WHAT?"
                       ├── Speaker: Confused tones
                       └── Antenna: Tilting motion
}}}

===== AI Integration Features =====

==== Voice Recognition System ====
* **Processor**: Raspberry Pi (TensorFlow Lite or Vosk)
* **Commands**: Natural language processing
* **Examples**:
  - "ARGOS, move forward"
  - "ARGOS, stop"
  - "ARGOS, what do you see?"
  - "ARGOS, are you okay?"
* **Response**: LCD text + appropriate emotion

==== Computer Vision System ====
* **Camera**: OV7670 (VGA resolution)
* **Processor**: Raspberry Pi (OpenCV)
* **Capabilities**:
  - Object detection (person, car, animal)
  - Color tracking (follow colored objects)
  - Line following (path navigation)
  - Face detection (recognize owner)
* **Power consumption**: ~300mA during streaming

==== Vision Streaming Considerations ====
* **Continuous streaming**: High power consumption (~300-400mA)
* **Solution**: Event-based capture
  - Stream only when obstacle detected
  - Periodic snapshots (every 5 seconds)
  - Full streaming on voice command
* **Storage**: Save important frames to SD card

===== IMU Sensor Integration (From Drone Project) =====

==== Sensor Identification ====
* **Component**: 6-axis or 9-axis IMU (Inertial Measurement Unit)
* **Common types**: MPU6050 (6-axis) or MPU9250 (9-axis)
* **Axes measured**:
  - **Accelerometer**: X, Y, Z linear acceleration
  - **Gyroscope**: X, Y, Z rotational velocity
  - **Magnetometer**: X, Y, Z magnetic field (compass)

==== Applications for ARGOS ====

===== Terrain Detection =====
* **Tilt sensing**: Detect uphill/downhill slopes
* **Stability**: Prevent tipping over
* **Adaptive speed**: Slow down on inclines

===== Orientation Tracking =====
* **Compass heading**: Maintain direction
* **Turn detection**: Measure rotation angles
* **GPS-free navigation**: Dead reckoning

===== Impact Detection =====
* **Collision sensing**: Detect sudden impacts
* **Fall detection**: Emergency stop if flipped
* **Vibration analysis**: Motor health monitoring

==== Integration with ESP32 ====
* **Connection**: I2C interface (SDA, SCL)
* **Processing**: ESP32 reads raw data, sends to Raspberry Pi
* **Frequency**: 50-100Hz sampling rate
* **Data format**: JSON over UART

{{{code: lang="python" linenumbers="True"
# ESP32 sends to Raspberry Pi via UART
{
  "accel": {"x": 0.02, "y": -0.05, "z": 9.81},
  "gyro": {"x": 0.1, "y": -0.3, "z": 0.05},
  "mag": {"x": 25.3, "y": -12.1, "z": 45.2},
  "tilt": 5.2,  // degrees
  "heading": 178.5  // compass direction
}
}}}

===== CAN Bus Peripheral Control =====

==== MCP2515 CAN Controller Module ====
* **Purpose**: Organized peripheral communication
* **Advantages**:
  - Robust long-distance communication
  - Multiple devices on single bus
  - Error detection and correction
  - Real automotive protocol
* **Connection**: SPI to ESP32 or Raspberry Pi

==== Peripheral Organization ====
{{{code: lang="text" linenumbers="True"
CAN Bus Network:
├── Node 1: Motor Controller (L298N commands)
├── Node 2: Servo Controller (steering + antenna)
├── Node 3: LCD Display (messages)
├── Node 4: Sensor Hub (ultrasonic, IMU)
└── Node 5: Audio System (speaker control)
}}}

==== Benefits for ARGOS ====
* **Modularity**: Easy to add/remove components
* **Reliability**: Industrial-grade communication
* **Scalability**: Support up to 127 devices
* **Debugging**: Monitor all bus traffic
* **Real automotive experience**: Demonstrates IoT security concepts

===== Audio System from Recycled Phones =====

==== Component Extraction from Old Phones ====

===== Microphone Extraction =====
* **Location**: Usually near charging port or earpiece
* **Type**: MEMS microphone (3-4 pins)
* **Pins**: VCC, GND, DATA, CLK (digital) or VCC, GND, OUT (analog)
* **Desoldering**: Use hot air station or soldering iron
* **Testing**: Connect to ESP32 ADC, verify signal

===== Speaker Extraction =====
* **Location**: Bottom of phone (loudspeaker)
* **Type**: 8Ω or 4Ω dynamic speaker
* **Power**: 0.5-1W typical
* **Connection**: 2 wires (+ and -)
* **Amplification**: May need PAM8403 amplifier module

==== Integration with ESP32 ====
* **Microphone**: ADC pin or I2S digital input
* **Speaker**: DAC pin + amplifier module
* **Processing**: ESP32 handles audio, sends text to Raspberry Pi for AI
* **Latency**: Low (real-time audio response)

===== Chassis Space Management =====

==== Components to Fit ====
* **Controllers**: ESP32 + Raspberry Pi Zero W
* **Batteries**: 12V (10900mAh), Powerbank (10000mAh), AA x4
* **Modules**: L298N, MCP2515, LCD, IMU
* **Motors**: 2x JGA25 + 2x servos
* **Sensors**: HC-SR04, camera, microphone, speaker

==== Space Optimization Strategies ====

===== Vertical Stacking =====
{{{code: lang="text" linenumbers="True"
Layer 1 (Bottom): 12V Battery + Motors
Layer 2 (Middle): L298N + Motor wiring
Layer 3 (Upper): Raspberry Pi + ESP32 + Sensors
Layer 4 (Top): Camera + Antenna servo
}}}

===== External Mounting =====
* **Powerbank**: Side-mounted with velcro straps
* **AA battery holder**: Rear-mounted externally
* **LCD display**: Front-facing mount
* **Antenna servo**: Top center mount

===== Modular Design =====
* **Removable trays**: 3D printed component holders
* **Cable management**: Zip ties and cable guides
* **Access panels**: Easy battery replacement
* **Weight distribution**: Heavier components low and centered

===== Alternative: Shared Power for Microcontrollers =====
If chassis space is critically limited, ESP32 and Raspberry Pi can share the powerbank:

==== Shared Powerbank Configuration ====
* **Powerbank**: 10,000mAh → USB outputs (2 ports)
  - Port 1: Raspberry Pi (5V 2A)
  - Port 2: ESP32 via buck converter (5V → 3.3V)
* **Advantages**: One less battery, simplified design
* **Disadvantages**: Shorter runtime, single point of failure
* **Runtime**: ~6-8 hours (both systems active)
