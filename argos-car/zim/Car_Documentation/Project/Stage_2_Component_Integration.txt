====== Stage 2 Component Integration ======
Created Friday 07 February 2026

===== Overview =====

This document consolidates all technical decisions, component distribution strategy, and integration plan for ARGOS Stage 2: Dual-Brain Architecture implementation.

**Purpose:** Transform ARGOS from single-controller (Raspberry Pi only) to dual-controller system (Raspberry Pi + ESP32) for improved performance, distributed processing, and scalability.

---

===== Architecture Philosophy =====

==== Symbiotic Dual-Brain System ====

**Raspberry Pi Zero W** = Central Nervous System
* High-level decision making and AI processing
* Computer vision and voice recognition
* Complex navigation logic
* Motor control (time-critical operations)

**ESP32** = Peripheral Nervous System  
* Real-time sensor processing (non-critical)
* User interface management (LCD display)
* Orientation tracking (IMU sensor)
* Offload tasks from Raspberry Pi to free CPU resources

**Design Goal:** Each controller handles tasks it does best, creating efficient symbiotic relationship.

---

===== Component Distribution Strategy =====

==== Raspberry Pi Zero W Responsibilities ====

**AI & Computer Vision (High CPU Load):**
* 📷 **Camera OV7670** - Object detection using MobileNet Lite
* 🎤 **Voice Recognition** - Vosk lightweight model via Bluetooth headset
* 🧠 **Navigation Logic** - Path planning, decision trees
* 🤖 **System Coordination** - Send high-level commands to ESP32

**Time-Critical Control (Low Latency Required):**
* 🚗 **Motor Control** - L298N controller (IN1-4, ENA, ENB)
* 📡 **Ultrasonic Sensor HC-SR04** - Obstacle detection and emergency brake
* ⚡ **Differential Drive Logic** - Turning maneuvers

**Communication:**
* 🔗 **UART to ESP32** - Send display messages and receive sensor data
* 📱 **Bluetooth A2DP** - Audio headset for voice commands and responses

**Power Source:**
* 🔋 Powerbank 5V 10,000mAh
* Estimated consumption: 300-400 mA
* Runtime: 10-15 hours

---

**User Interface (Medium CPU Load):**
* 🖥️ **LCD 1602 Display** - System status messages, emotional states

**Orientation & Navigation Support:**
* 🧭 **IMU Sensor (MPU6050)** - Maintain straight line direction
* 📊 **Tilt detection** - Terrain awareness
* 🔄 **Heading correction** - Auto-adjust steering drift

**Communication:**
* 🔗 **UART to Raspberry Pi** - Receive commands, send IMU data

**Power Source:**

===== Power System Architecture =====

==== Independent Power Sources ====

**Source 1: Raspberry Pi + Ultrasonic Sensor**
```
Powerbank 5V 10,000mAh
├── Raspberry Pi Zero W (via micro-USB)
└── HC-SR04 Sensor (via USB cable cut + jumpers)
```

**Source 2: ESP32 + Display + IMU**
```
4x AA Batteries (4.8V) via USB cable cut
├── ESP32 (via VIN pin + jumpers or soldered USB Type-C connector)
├── LCD 1602 (powered by ESP32 5V out)
└── IMU MPU6050 (powered by ESP32 3.3V out)
```

**Source 3: Motors**
```
12V Battery 10,900mAh
├── L298N Controller (+12V input)
└── 2x JGA25 12V DC Gearmotors
```

**Common Ground:**
* All three power systems share GND reference
* GND connected via Raspberry Pi pins 6, 9, 14

**Advantages:**
* ✅ Electrical isolation prevents cross-contamination
* ✅ Motor power spikes don't affect logic circuits
* ✅ Each system operates at optimal voltage
* ✅ Easy to troubleshoot individual power issues

===== Communication Protocol =====

==== UART Serial Communication ====

**Physical Connection:**
```
Raspberry Pi GPIO 14 (Pin 8)  TX → ESP32 GPIO 16 (RX2)
Raspberry Pi GPIO 15 (Pin 10) RX ← ESP32 GPIO 17 (TX2)
Common GND between both controllers
```

**Note:** GPIO 15 (Pin 10) currently occupied by ENA. Solution: Move ENA to GPIO 12 (Pin 32) which also supports PWM.

**Protocol Specifications:**
* Baud Rate: 115200 bps
* Data Format: 8N1 (8 data bits, no parity, 1 stop bit)
* Latency: <1 ms (critical for real-time response)

**Message Format (JSON):**
```json
// Pi → ESP32 (Commands)
{
  "action": "display",
  "text": "MOVING FORWARD",
  "type": "status"
}

{
  "action": "query",
  "sensor": "imu",
  "data": "heading"
}

// ESP32 → Pi (Responses)
{
  "sensor": "imu",
  "heading": 178.5,
  "tilt": 2.3,
  "timestamp": 1234567890
}
```

**Why UART over Bluetooth/WiFi:**
* ✅ Minimal latency (<1 ms vs 50-100 ms for Bluetooth)
* ✅ Extremely reliable (wired connection)
* ✅ Simple implementation (only 2 wires + GND)
* ✅ Doesn't consume wireless resources
* ✅ Perfect for real-time coordination

==== ESP32 GPIO Usage ====

|| Component | ESP32 GPIO | Function | Notes |
||-----------|------------|----------|-------|

**Note on I2C Sharing:**
* LCD 1602 (I2C address: 0x27 or 0x3F typical)
* IMU MPU6050 (I2C address: 0x68 or 0x69)
* Different addresses allow both devices on same bus
* Significant pin savings (2 pins control 2 devices)

===== Power Connection Details =====

==== ESP32 Power Solution ====

**Option A: Jumper Cables (RECOMMENDED)**
```
USB cable cut (from AA batteries)
├── Red wire (+4.8V) → Jumper male → ESP32 VIN pin
└── Black wire (GND)  → Jumper male → ESP32 GND pin
```

**Advantages:**
* ✅ Simple and reliable
* ✅ Standard jumper wires support 2-3A (ESP32 needs max 0.5A)
* ✅ 4-6x safety margin
* ✅ No complex soldering required

**Option B: USB Type-C Connector**
```
USB cable cut → Solder to Type-C connector → Plug into ESP32
```

**Implementation:**
* Cut Type-C cable
* Identify only 2 wires needed: VCC (red/orange) and GND (black)
* Ignore data wires (green/white)
* Solder to battery cable

**Advantages:**
* ✅ Clean, professional look
* ✅ Easy to disconnect ESP32
* ❌ More work for same result

**Power Requirements:**
* ESP32 typical consumption: 100-150 mA
* ESP32 peak consumption: ~500 mA
* AA batteries capacity: 2000-2500 mAh
* Expected runtime: 6-8 hours continuous operation

===== IMU Sensor Integration =====

==== Component: MPU6050 (6-axis IMU) ====

**Specifications:**
* 3-axis accelerometer + 3-axis gyroscope
* I2C interface (2 wires: SDA, SCL)
* I2C address: 0x68 (default) or 0x69 (alternative)
* Operating voltage: 3.3V or 5V (both supported)

**Primary Functions:**

1. **Heading Maintenance (Direction Correction)**
```python
# Auto-correct to maintain straight line
target_heading = 0  # North reference
current_heading = imu.get_heading()

if current_heading > target_heading + 5:
    # Drifting right, adjust left
    adjust_motors(left=100%, right=85%)
elif current_heading < target_heading - 5:
    # Drifting left, adjust right
    adjust_motors(left=85%, right=100%)
```

2. **Tilt Detection (Terrain Awareness)**
```python
tilt_angle = imu.get_tilt()
if tilt_angle > 15:
    print("WARNING: Steep incline detected")
    reduce_speed(50%)  # Prevent tipping
```

3. **Position Estimation (Dead Reckoning)**
```python
# Estimate distance traveled without GPS
distance = integrate(acceleration, time)
position = integrate(velocity, time)
```

**Integration Priority:**
* ⏳ **Stage 2, Phase C** (not critical for basic functionality)
* ✅ Add after motor control and display are validated
* 🎯 Enhances navigation quality significantly

**Why Include IMU:**
* ✅ Dramatically improves straight-line movement
* ✅ Enables predictable turning angles
* ✅ Critical for hackeo demonstration (sensor disable shows drift)
* ✅ Low cost (~$3-5) and easy integration (I2C shared with LCD)
* ✅ Makes project more impressive in presentation

===== Hackeo Demonstration Strategy =====

==== Vulnerabilities to Exploit (Without CAN Bus) ====

**Original Plan:** CAN Bus injection attack (Jeep Cherokee style)  
**Status:** ❌ Removed due to complexity, cost, and time constraints

**New Attack Vectors:**

===== Attack Vector 1: GPIO Command Injection =====
```
Exploit: Direct manipulation of motor control signals
Method: Inject malicious commands through software vulnerability

Normal Operation:
User voice command → Pi processes → Motors respond safely

Hacked Operation:
Attacker bypasses authentication → Sends direct GPIO commands
→ Motors ignore safety checks → Full speed ahead
```

===== Attack Vector 2: Sensor Data Poisoning =====
```
Exploit: Feed false data to navigation system
Method: Man-in-the-middle attack on sensor readings

Normal Operation:
HC-SR04: "Obstacle at 20cm" → Pi: "STOP MOTORS"
IMU: "Drifting right 8°" → Pi: "Correct left"

Hacked Operation:
Attacker intercepts sensor data → Replaces with fake data
HC-SR04: "No obstacle" (false) → Pi: "Continue" → CRASH
IMU: "Heading 0°" (false) → Pi: "No correction" → DRIFT
```

===== Attack Vector 3: Privilege Escalation =====
```
Exploit: Gain root access to Raspberry Pi
Method: SSH brute force or software vulnerability

Attack Chain:
1. Scan network for Raspberry Pi (port 22)
2. Attempt SSH connection with common passwords
3. Gain shell access
4. Escalate to root privileges
5. Disable safety systems
6. Control motors directly
```

===== Attack Vector 4: Bluetooth Hijacking =====
```
Exploit: Take over voice command channel
Method: Bluetooth pairing vulnerability

Normal Operation:
Authorized headset → Voice commands → Pi executes safely

Hacked Operation:
Attacker pairs malicious Bluetooth device
→ Sends fake voice commands
→ "ARGOS, full speed ahead"
→ "ARGOS, disable obstacle detection"
```

===== Critical Decisions Summary =====

==== Architecture Decisions ====
* ✅ **Dual-controller system adopted** (Raspberry Pi + ESP32)
* ✅ **UART communication chosen** over Bluetooth/WiFi (latency priority)
* ✅ **Three independent power sources** (motors, Pi, ESP32)
* ❌ **CAN Bus removed** (complexity, cost, time constraints)

==== Component Placement Decisions ====
* ✅ **Motors on Pi** (time-critical, emergency brake priority)
* ✅ **HC-SR04 on Pi** (for now, may migrate to ESP32 later)
* ✅ **LCD on ESP32** (frees Pi CPU for AI processing)
* ✅ **IMU on ESP32** (real-time orientation, non-critical latency)
* ✅ **Camera on Pi** (GPU acceleration, CSI port advantage)
* ✅ **Voice via Bluetooth headset** (eliminates mic+speaker complexity)

==== Technical Decisions ====
* ✅ **I2C bus sharing** (LCD + IMU on same bus, different addresses)
* ✅ **Move ENA to GPIO 12** (frees GPIO 15 for UART RX)
* ✅ **Jumper wires for ESP32 power** (simpler than Type-C connector)
* ✅ **TensorFlow Lite models** (optimize for Pi Zero W limited resources)
* ✅ **Stage-based integration** (validate each component before adding next)

==== Deferred Decisions (Pending Clarification) ====
* ⏳ **Camera interface method** (CSI vs GPIO - depends on hardware)
* ⏳ **LCD I2C availability** (need to verify module present)
* ⏳ **Attack vector choice** (choose 1-2 vulnerabilities for demonstration)

===== Risk Mitigation =====

==== Identified Risks & Solutions ====

**Risk 1: UART Communication Failure**
* Symptom: ESP32 and Pi cannot communicate
* Mitigation: Test with simple echo program first, verify common GND
* Backup: Use I2C communication as alternative (slower but works)

**Risk 2: Power Insufficiency**
* Symptom: ESP32 brownouts, random resets
* Mitigation: Measure actual current draw, upgrade batteries if needed
* Backup: Power ESP32 from powerbank temporarily

**Risk 3: GPIO Pin Shortage**
* Symptom: Not enough pins for all components
* Mitigation: Prioritize I2C components (shared bus), use multiplexers if needed
* Backup: Remove non-critical components (antenna servo already removed)

**Risk 4: Pi Zero W Performance**
* Symptom: System lags, frame drops, slow response
* Mitigation: Use TensorFlow Lite, reduce camera resolution, optimize code
* Backup: Offload more processing to ESP32

**Risk 5: Integration Complexity**
* Symptom: Too many components break existing functionality
* Mitigation: Stage-based approach, test after each addition
* Backup: Roll back to previous working state (Git version control)
