====== Tercer Dia ======
Created Saturday 04 January 2025

===== Problems in Paradise =====

The Raspberry Pi Zero W suddenly stopped responding to SSH connections. The Pi powers on correctly (green LED active), SD card appears intact, but no network connectivity via WiFi or USB gadget interface.

==== Symptoms Observed ====
* ssh raspi → No route to host
* ssh pi@192.168.0.2 (iotrac) → No route to host  
* ssh pi@10.247.104.100 (KironWifi) → Connection timeout
* ssh pi@192.168.7.2 (USB gadget) → Destination Host Unreachable
* Raspberry Pi LED: Green light ON (system powered and booting)
* SD card: No physical damage, files accessible from PC

===== Exhaustive SD Card Analysis =====

==== Step 1: Device Detection ====
{{{code: lang="sh" linenumbers="True"
# Identify SD card device and partitions
lsblk -o NAME,SIZE,TYPE,FSTYPE,LABEL,MOUNTPOINT,MODEL
sudo blkid /dev/sda1 /dev/sda2

# Results:
# sda1: bootfs (FAT) - already mounted at /media/kiron/bootfs
# sda2: rootfs (ext4) - needs manual mounting
}}}

==== Step 2: Mount and Initial Verification ====
{{{code: lang="sh" linenumbers="True"
# Mount root partition read-only for safe inspection
sudo mkdir -p /mnt/pi-root
sudo mount -o ro /dev/sda2 /mnt/pi-root

# Boot partition already available at /media/kiron/bootfs
}}}

==== Step 3: Boot Configuration Analysis ====
{{{code: lang="sh" linenumbers="True"
# Check USB gadget and UART configuration
grep -E '^(enable_uart|dtoverlay)=' /media/kiron/bootfs/config.txt
cat /media/kiron/bootfs/cmdline.txt
ls -l /media/kiron/bootfs/ssh

# Results found:
# ✅ dtoverlay=dwc2,g_ether (USB gadget enabled)
# ✅ modules-load=dwc2,g_ether in cmdline.txt
# ❌ enable_uart=1 MISSING
# ❌ ssh file MISSING (SSH disabled)
}}}

==== Step 4: NetworkManager Profiles Verification ====
{{{code: lang="sh" linenumbers="True"
# Check NetworkManager connection profiles
sudo ls -l /mnt/pi-root/etc/NetworkManager/system-connections/
sudo cat /mnt/pi-root/etc/NetworkManager/system-connections/iotrac.nmconnection
sudo cat /mnt/pi-root/etc/NetworkManager/system-connections/KironWifi.nmconnection
sudo cat /mnt/pi-root/etc/NetworkManager/system-connections/usb-gadget.nmconnection

# Results:
# ✅ All profiles exist with correct permissions (600 root:root)
# ✅ KironWifi: priority 100, static IP 10.247.104.100
# ✅ iotrac: priority 50, static IP 192.168.0.2  
# ⚠️ usb-gadget: interface-name=eth0 (should be usb0)
}}}

==== Step 5: Service Conflicts Detection ====
{{{code: lang="sh" linenumbers="True"
# Check for conflicting network services
sudo ls -l /mnt/pi-root/etc/systemd/system/multi-user.target.wants/NetworkManager.service
sudo ls -l /mnt/pi-root/etc/systemd/system/multi-user.target.wants/ | grep -E 'dhcpcd|wpa'

# Results:
# ✅ NetworkManager.service enabled
# ❌ wpa_supplicant.service ENABLED (conflicts with NetworkManager)
# ✅ dhcpcd not enabled
}}}

==== Step 6: WiFi Blocking Investigation ====
{{{code: lang="sh" linenumbers="True"
# Check for WiFi blocking configurations
sudo find /mnt/pi-root/etc -name "*rfkill*" -o -name "*wifi*"
sudo cat /mnt/pi-root/etc/modprobe.d/rfkill_default.conf

# Critical finding:
# ❌ options rfkill default_state=0 (WiFi BLOCKED by default)
}}}

==== Step 7: Additional Conflict Detection ====
{{{code: lang="sh" linenumbers="True"
# Search for additional network configuration conflicts
sudo find /mnt/pi-root/etc -name "*.conf" | grep -iE "(network|wifi|dhcp)"
sudo ls -la /mnt/pi-root/etc/systemd/system/ | grep -iE "(network|wifi|dhcp|wpa)"

# Found potential conflicts:
# ⚠️ dbus-fi.w1.wpa_supplicant1.service (D-Bus alias for wpa_supplicant)
# ✅ dhcp/dhclient.conf exists but dhcpcd service disabled
# ✅ systemd-networkd configs nullified (/dev/null)
}}}

===== Three Major Problems Identified =====

==== Problem 1: SSH Service Disabled ====
* **Issue**: Missing `/boot/ssh` file
* **Impact**: SSH daemon not starting on boot
* **Severity**: Critical - no remote access possible

==== Problem 2: WiFi Radio Blocked by Default ====
* **Issue**: `rfkill default_state=0` in `/etc/modprobe.d/rfkill_default.conf`
* **Impact**: WiFi hardware disabled at kernel level
* **Severity**: Critical - prevents all WiFi connections

==== Problem 3: wpa_supplicant Service Conflict ====
* **Issue**: `wpa_supplicant.service` enabled alongside NetworkManager
* **Impact**: Two network managers competing for WiFi control
* **Severity**: High - causes connection instability and failures

===== Solutions Applied =====

==== Fix 1: Enable SSH Service ====
{{{code: lang="sh" linenumbers="True"
# Remount in write mode and enable SSH
sudo umount /mnt/pi-root
sudo mount /dev/sda2 /mnt/pi-root
sudo touch /media/kiron/bootfs/ssh
}}}

==== Fix 2: Enable WiFi Hardware ====
{{{code: lang="sh" linenumbers="True"
# Unblock WiFi radio by default
sudo sed -i 's/default_state=0/default_state=1/' /mnt/pi-root/etc/modprobe.d/rfkill_default.conf
}}}

==== Fix 3: Remove wpa_supplicant Conflicts ====
{{{code: lang="sh" linenumbers="True"
# Disable wpa_supplicant service
sudo rm /mnt/pi-root/etc/systemd/system/multi-user.target.wants/wpa_supplicant.service
sudo rm /mnt/pi-root/etc/systemd/system/dbus-fi.w1.wpa_supplicant1.service
}}}

==== Fix 4: Simplify Connection Profiles ====
{{{code: lang="sh" linenumbers="True"
# Remove unnecessary connections to prevent conflicts
sudo rm /mnt/pi-root/etc/NetworkManager/system-connections/internet.nmconnection
sudo rm /mnt/pi-root/etc/NetworkManager/system-connections/usb-gadget.nmconnection
sudo rm /mnt/pi-root/etc/systemd/system/multi-user.target.wants/usb-gadget-autoconnect.service
}}}

==== Fix 5: Enable Serial Console (Backup Access) ====
{{{code: lang="sh" linenumbers="True"
# Add UART support for emergency console access
echo 'enable_uart=1' | sudo tee -a /media/kiron/bootfs/config.txt
}}}

===== Final Resolution: Reflash Required =====

After exhaustive troubleshooting and applying all identified fixes, the Raspberry Pi still failed to establish network connectivity. Despite correcting all known issues (SSH enabled, WiFi unblocked, wpa_supplicant conflicts removed), the Pi did not appear in network scans and showed no signs of attempting network connections.

==== Decision Point ====
* All software-level fixes applied without success
* Hardware appeared functional (LED indicators normal)
* SD card integrity verified and corrections applied
* No response to ping or SSH on any configured IP addresses
* ARP table showed no Pi presence on any network

===== Root Cause Analysis =====

==== What Went Wrong ====
Despite comprehensive troubleshooting that identified and corrected multiple legitimate issues:
1. SSH service disabled
2. WiFi hardware blocked by rfkill
3. wpa_supplicant service conflicts
4. Excess connection profiles causing confusion

The system still failed to establish connectivity, suggesting:
* **Deep system corruption** not visible in file-level inspection
* **Hidden configuration conflicts** in binary/cache files
* **Kernel-level issues** not detectable through standard diagnostics
* **NetworkManager internal state corruption** requiring clean reinstallation

==== Why Reflash Succeeded ====
* **Clean system state** - no hidden conflicts or corrupted caches
* **Fresh NetworkManager installation** - no internal state corruption
* **Default boot process** - no race conditions or startup conflicts
* **Clean kernel modules** - WiFi drivers loaded without conflicts

===== Lessons Learned =====

==== When to Consider Reflash ====
1. **Hardware confirmed functional** (Pi boots, LEDs normal)
2. **All software fixes applied** but problem persists
3. **No network presence detected** (ARP, ping, scan failures)
4. **Time constraints** require quick resolution
5. **System shows signs of deep corruption** beyond file-level fixes

==== Diagnostic Hierarchy ====
1. **File-level configuration** (what we did first)
2. **Service and permission issues** (systemd, NetworkManager)
3. **Hardware blocking** (rfkill, firmware)
4. **Deep system conflicts** (cache, binary state)
5. **Complete system refresh** (reflash as last resort)

===== Motor Controller Diagnosis Session =====

After successful connectivity restoration, focused on resolving the persistent motor control issue where L298N controller shows power (LED ON) but motors remain unresponsive.

==== Systematic Elimination Process ====

===== ❌ Problems ELIMINATED (Not the cause) =====

==== ENA/ENB Jumper Configuration ====
* **Initial hypothesis**: Missing ENA/ENB GPIO connections causing motor disable
* **Test method**: Created motor_test_simple.py using only IN1-IN4 pins with jumpers enabled
* **Result**: Motors still unresponsive with jumpers connected (always-enabled mode)
* **Conclusion**: ENA/ENB control method not the issue

==== Jumper Cable Integrity ====
* **Initial hypothesis**: Cut and reconnected jumper cables have intermittent connections
* **Test method**: Executed blinky.py using same jumper cables for LED control
* **Result**: LEDs responded perfectly to GPIO commands through same cables
* **Conclusion**: Cable continuity and GPIO signal transmission confirmed functional

==== Power Supply Issues ====
* **Initial hypothesis**: Insufficient power or voltage drops
* **Test method**: Multimeter verification of powerbank output and cable endpoints
* **Result**: 5V confirmed stable, L298N LED consistently ON
* **Conclusion**: Power delivery to controller confirmed adequate

==== Direct Hardware Test ====
* **Initial hypothesis**: GPIO signal levels insufficient for L298N input
* **Test method**: Direct connection of IN1-IN4 pins to 5V/GND from powerbank
* **Result**: No motor response even with direct 5V signal input
* **Conclusion**: Problem exists beyond GPIO signal levels

===== ⚠️ Remaining POSSIBLE Causes =====

==== L298N Controller Internal Failure ====
* **Symptoms**: LED indicates power but no motor output response
* **Possibility**: Power circuitry functional but H-bridge drivers damaged
* **Evidence**: No response to direct 5V input on control pins
* **Next steps**: Replace L298N module for definitive test

==== Motor Physical Failure ====
* **Symptoms**: Both motors completely unresponsive 
* **Possibility**: Internal motor windings damaged or mechanical seizure
* **Evidence**: No movement, sound, or resistance change during tests
* **Next steps**: Direct powerbank connection to motor terminals (bypass L298N)

==== Wiring Configuration Error ====
* **Symptoms**: Correct pin assignments but no functional response
* **Possibility**: Motor polarity, ground reference, or connection sequence issue
* **Evidence**: All logical connections verified but system non-functional
* **Next steps**: Complete rewiring with fresh components
